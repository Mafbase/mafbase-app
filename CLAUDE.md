# Flutter / Dart

Сначала пробуй `fvm flutter` / `fvm dart`. Если нет прав на выполнение — проверь что версия `flutter --version` совпадает с указанной в `.fvmrc`, и если да — используй `flutter` / `dart` напрямую.

# Правила разработки фичи

При получении задачи на новую фичу или значительное UI-изменение, следуй этому процессу:

## Этап 1 — Параллельный запуск агентов

Запусти одновременно (в одном сообщении) следующих агентов:

1. **designer** — прорабатывает визуальный дизайн фичи, задаёт пользователю вопросы о внешнем виде, согласовывает детали
2. **Explore** — исследует кодовую базу: находит связанные файлы, существующие паттерны, модели данных, роутинг

Дождись результатов обоих агентов перед переходом к следующему этапу.

## Этап 2 — Планирование

Войди в план-режим (`EnterPlanMode`) и составь план реализации на основе:
- Согласованного дизайна от агента designer
- Найденных паттернов и файлов от агента Explore
- Правил из `AGENTS.md` (локализация, DI, структура виджетов, proto)

План должен содержать:
- Список файлов для создания/изменения
- Порядок реализации (модели → bloc → UI → тесты)
- Какие зависимости нужны (репозитории, хранилища, proto-модели)

## Этап 3 — Реализация

После одобрения плана пользователем, реализуй фичу в таком порядке:

1. **Модели и данные** — domain models, repository interfaces, data layer
2. **Бизнес-логика** — Bloc/Event/State
3. **UI** — страница и виджеты по согласованному дизайну
4. **Локализация** — строки в `app_ru.arb` + `fvm flutter gen-l10n`
5. **Роутинг** — добавить маршрут в `router.dart`
6. **Проверка** — `fvm flutter analyze`

## Этап 4 — Финализация

- Запусти golden-тесты для новых страниц (скилл `/golden-test`)
- Убедись что `fvm flutter analyze` проходит без ошибок

# Обработка ошибок в Bloc

Не добавляй try/catch с обработкой ошибок в каждый bloc event handler. Ошибки обрабатываются глобальным BlocObserver. В handler нужно только сбрасывать состояние загрузки (isLoading: false) через try/finally.

- Когда в handler ставится `isLoading: true`, нужен try/finally для сброса загрузки
- Если isLoading не используется — try/catch не нужен

# UI-компоненты

Всегда используй готовые компоненты проекта вместо стандартных Flutter-виджетов:

- `CustomButton` вместо `ElevatedButton` / `TextButton` / `OutlinedButton`
- `CustomDropdown` вместо стандартного `DropdownButton`

Компоненты находятся в `lib/common/widgets/`.

# Ведение CHANGELOG

При любых пользовательски-видимых изменениях (фичи, баг-фиксы, UI-изменения) обновляй `CHANGELOG.md`:

1. Если в начале файла уже есть секция `## [UNRELEASED]` — добавь новые пункты в неё
2. Если секции нет — создай её в самом начале файла:
   ```markdown
   ## [UNRELEASED]

   - Описание изменения
   ```
3. Правила описания — те же что в скилле `/release`: русский язык, только видимые пользователю изменения, глаголы "Добавлена/Исправлена/Изменена"
4. **Не** пиши дату и версию — это делает скилл `/release`
5. **Не** обновляй CHANGELOG для внутренних изменений (рефакторинг, CI/CD, тесты)

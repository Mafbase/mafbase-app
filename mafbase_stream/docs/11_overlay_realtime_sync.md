# Задача 11: Реализация overlay view и real-time синхронизация

## Цель

Используя механизм из задачи 10, реализовать **конкретную** overlay view с плашками игроков, логотипом турнира, плашкой раунда и ротацией спонсоров. Подключить real-time канал состояния игры от Mafbase: ведущий меняет статус игрока на своём устройстве → стрим на штативном телефоне обновляет плашку с анимацией ≤2 секунд.

Вёрстка ведётся нативно (Compose на Android, SwiftUI на iOS) — это позволяет работать над оверлеем независимо от пайплайна стриминга.

## Связь с BRD

- §4.3.1 F3.1–F3.5: содержимое оверлея.
- §4.3.2 F3.6, F3.7, F3.9: оверлей всегда виден, анимация перехода ~300мс, полупрозрачность.
- §4.4 F4.1–F4.5: синхронизация состояния, ≤2 секунды задержки, оффлайн-очередь, восстановление.
- §10 Q2: «Существует ли в Mafbase real-time канал?» — должен быть выяснен до старта.

## Содержание

1. Унифицированная Dart-модель `OverlayState`: список игроков (имя, статус, роль), номер раунда, логотип турнира, список спонсоров с интервалом ротации.
2. Нативная overlay view:
   - **Android** — `OverlayView.kt` на Jetpack Compose: фиксированный layout по периметру кадра, плашки игроков, плашка раунда, логотипы.
   - **iOS** — `OverlayView.swift` на SwiftUI с тем же layout.
   - Бэкенд view — `OverlayState`, который обновляется через метод `setState(state)`.
3. Регистрация view в плагине через механизм задачи 10: при старте стрима приложение создаёт overlay-view и передаёт её handle в `MafbaseStream.attachOverlayView(...)`.
4. Анимация перехода ~300мс (F3.7) — нативные animation API (Compose `animate*AsState`, SwiftUI `withAnimation`). После завершения кадра анимации view дёргает `MafbaseStream.invalidateOverlay()`, чтобы плагин перезалил текстуру (или плагин подписан на `Choreographer` / `CADisplayLink` пока идёт анимация).
5. Ротация спонсоров — таймер внутри view, дёргающий `invalidate` на каждой смене.
6. Real-time sync (`lib/src/overlay_sync.dart`):
   - подписка на канал Mafbase (WebSocket/SSE — уточнить с бэкендом до старта);
   - событие `tournamentGameStateChanged` → адаптер модели игры → `OverlayState` → `setState` нативной view;
   - на стороне ведущего убедиться, что его действия публикуются в тот же канал.
7. Оффлайн-поведение (F4.4–F4.5): при потере связи view продолжает показывать последнее состояние; при восстановлении — догоняет очередь по timestamp.
8. Замер end-to-end задержки ≤2 сек (F4.3): timestamp действия ведущего vs timestamp применения в overlay view, выводится в лог.

## Критерий приёмки

- Сценарий: 2 устройства подключены к одному турниру в Mafbase. На устройстве A (ведущий) меняется статус игрока «жив → убит». На устройстве B (стример) идёт стрим в локальный nginx. В VLC через ≤2 секунды видна обновлённая плашка с анимацией.
- 30-секундный стрим без artifacts: плашки не «дёргаются», текст не дрожит, переходы плавные.
- Логотип спонсора периодически меняется.
- При выключении Wi-Fi на стримящем устройстве на 30 сек: плашки не пропадают, RTMP реконнектится (см. задачу 13), после восстановления — статусы догоняются.
- Логи показывают timestamp delta в пределах целевого SLA.

## Артефакты

- `android/.../OverlayView.kt` — Compose-вёрстка оверлея.
- `ios/.../OverlayView.swift` — SwiftUI-вёрстка оверлея.

- `lib/src/overlay_sync.dart` — Dart-клиент канала и адаптер `MafbaseGameState → OverlayState`.

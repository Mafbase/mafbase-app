# Задача 13: Адаптивный битрейт и переподключение

## Цель

Стрим адаптируется к качеству сети (битрейт снижается при загруженности и поднимается при улучшении) и переживает временные обрывы соединения с RTMP-сервером.

## Связь с BRD

- §4.2 F2.5: адаптивный битрейт.
- §4.2 F2.7: автореконнект, минимум 3 попытки с увеличивающимся интервалом.
- §5.2 NF2.2: при потере сети не падает.
- §9 R3: митигация для нестабильной мобильной сети.

## Содержание

1. В C++ ядре — `BitrateController`:
   - входы: длина очереди `av_interleaved_write_frame`, RTT (если доступно), процент дропнутых кадров;
   - выход: целевой битрейт, переключаемый через MediaCodec/VideoToolbox API.
2. Шаги: 100% → 75% → 50% → 33% (от выбранного preset). Возврат вверх — только после стабильного периода.
3. Реконнект: при ошибке записи или закрытии сокета — exponential backoff (1с → 2с → 4с, далее cap 8с), до 3 попыток, потом — событие `streamFailed` в Dart.
4. State machine: `streaming → reconnecting → streaming|stopped`.
5. Захват кадров **не останавливается** во время реконнекта — буферизуется ограниченная очередь (например, 2 секунды), при превышении дропаются старые non-keyframe.
6. Логирование событий через MethodChannel (см. задачу 18).

## Критерий приёмки

- Тест с `tc qdisc` (Linux) или Network Link Conditioner (iOS): искусственно ограниченная полоса 1 Мбит → битрейт стрима опускается, обрывов нет.
- Полная блокировка сети на 10–20 сек: после восстановления стрим продолжается без перезапуска приложения.
- Полная блокировка >30 сек (выходим за лимит реконнектов): в Dart прилетает событие `streamFailed`, экран стрима показывает уведомление, RTMP-сессия в `stopped`.
- 30-минутная сессия с периодическими подёргиваниями сети (mock): без crash, без зависшего UI.

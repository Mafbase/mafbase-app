## 1. Контекст и проблема

### 1.1 Текущая ситуация

Mafbase — мобильное приложение для проведения и учёта турниров по Мафии. Организаторы оплачивают турниры через приложение и получают инструменты для ведения игр, учёта статистики, управления участниками.

На текущий момент трансляция турниров требует от организатора отдельного сетапа: ноутбук с OBS Studio, отдельная камера или захват с телефона, ручная настройка сцен, плашек игроков, ввод информации в реальном времени. Это создаёт высокий порог входа: нужно техническое оборудование, оператор стрима, время на настройку.

### 1.2 Проблема

Большинство региональных и любительских турниров **не транслируются** или транслируются с минимальным качеством (одна камера без оверлеев, без статусов игроков), потому что у организаторов нет ресурсов на полноценный стрим-сетап. Это снижает охват турниров, ограничивает возможности для спонсоров и сообщества.

### 1.3 Решение

Встроить в Mafbase возможность вести профессионально выглядящий стрим **только с одного телефона** на штативе. Приложение само:

- Захватывает видео с камеры и звук с микрофона
- Накладывает оверлей с актуальными статусами игроков, логотипами, текущим раундом
- Отправляет готовый поток в произвольный RTMP-сервер (как OBS) — целевая платформа в первую очередь VK Видео, но без жёсткой привязки
- Синхронизирует статусы игроков с действиями ведущего, который управляет игрой со второго устройства

В результате организатор получает «OBS в кармане» без необходимости докупать оборудование или нанимать оператора.

## 2. Бизнес-цели и метрики

### 2.1 Бизнес-цели

- **B1.** Повысить ценность платных турниров за счёт встроенной трансляции — увеличить готовность организаторов оплачивать турниры через Mafbase.
- **B2.** Расширить присутствие Mafbase в публичном пространстве через трансляции турниров с брендированием.
- **B3.** Снизить барьер входа в трансляции для региональных и небольших клубов.

### 2.2 Метрики успеха (для оценки после запуска)

| Метрика | Целевое значение (через 3 мес после запуска) |
|---|---|
| Доля турниров с активной трансляцией | ≥ 15% от платных турниров |
| Количество стрим-сессий за месяц | ≥ 50 |
| Средняя длительность стрим-сессии | ≥ 90 минут |
| Стабильность стрима (доля сессий без обрывов > 30 сек) | ≥ 90% |
| Удовлетворённость организаторов фичей (опрос/NPS) | ≥ 7/10 |

> **ДОПУЩЕНИЕ:** Целевые значения метрик заданы на глаз. Перед запуском стоит свериться с реальной базой пользователей Mafbase и историей турниров, чтобы скорректировать.

## 3. Целевые пользователи

### 3.1 Основная роль: Стример (Streamer)

Организатор или назначенный им оператор. Управляет камерой и стримом со штатива.

**Контекст:**
- Телефон закреплён на штативе сверху над игровым столом, объектив направлен вниз
- Снимает обзор стола с игроками
- Не имеет физического доступа к управлению статусами в момент игры (руки заняты, телефон далеко)
- Задача: запустить стрим, удостовериться что всё работает, не трогать телефон до конца игры

**Технический уровень:** средний. Знает что такое RTMP URL и stream key, способен скопировать их из VK Видео или другого сервиса.

### 3.2 Основная роль: Ведущий с управлением (Game Master)

Ведёт игру, управляет статусами игроков с **отдельного устройства** (свой телефон/планшет). Изменения, которые он вносит, синхронизируются на сервер и через сервер попадают в стримящее приложение, обновляя оверлей.

**Контекст:**
- Уже использует Mafbase для ведения турнира
- Открывает экран управления игрой как обычно
- Не задумывается о том, что параллельно идёт стрим — стрим обновляется автоматически

> **ДОПУЩЕНИЕ:** Я предполагаю, что у ведущего УЖЕ есть в Mafbase экран управления текущей игрой со статусами игроков, и фича стрима только подключает оверлей к этому существующему состоянию. Если такого экрана нет, объём фичи увеличивается.

### 3.3 Вторичная роль: Зритель

Смотрит трансляцию на платформе (VK Видео и т.д.). В рамках этого BRD не является пользователем приложения, но потребляет результат фичи.

## 4. Функциональные требования

### 4.1 Запуск трансляции

- **F1.1.** Стример может начать настройку трансляции из экрана активного турнира в Mafbase.
- **F1.2.** Перед стартом стрима стример указывает:
  - RTMP server URL (произвольная строка, как в OBS)
  - Stream key (произвольная строка, как в OBS)
  - Разрешение/качество (preset из 2-3 вариантов)
  - Опцию локальной записи (вкл/выкл)
- **F1.3.** Приложение запоминает последние использованные настройки RTMP для удобства повторного запуска. Stream key хранится с учётом безопасности (Keystore/Keychain).
- **F1.4.** Перед стартом стрима приложение проверяет:
  - Наличие разрешений на камеру и микрофон (запрашивает если нет)
  - Доступность RTMP-сервера (тестовое подключение)
  - Достаточно ли свободного места при включённой записи
- **F1.5.** Стример нажимает «Начать стрим» — открывается полноэкранный режим трансляции, стрим стартует.

### 4.2 Захват и трансляция

- **F2.1.** Захват видео с основной камеры устройства в **горизонтальной ориентации**.

  > **ДОПУЩЕНИЕ:** Поскольку телефон на штативе сверху смотрит на стол, наиболее естественной ориентацией картинки будет landscape (горизонтальная). Если стол круглый и хочется вертикального кадра — нужно обсуждать. В MVP делаем горизонталь.

- **F2.2.** Захват аудио с встроенного микрофона устройства. Поддержка внешних микрофонов (Bluetooth, USB-C, Lightning) — bonus, не в MVP.
- **F2.3.** Кодирование видео в H.264, аудио в AAC.
- **F2.4.** Отправка потока на указанный RTMP/RTMPS сервер.
- **F2.5.** Адаптивный битрейт: при ухудшении сети битрейт снижается, чтобы избежать обрывов; при улучшении — восстанавливается.
- **F2.6.** Целевые параметры качества (presets):

  | Preset | Разрешение | FPS | Битрейт видео | Битрейт аудио |
  |---|---|---|---|---|
  | High | 1080p | 30 | 4500 kbps | 128 kbps |
  | Medium (default) | 720p | 30 | 2500 kbps | 128 kbps |
  | Low | 480p | 30 | 1000 kbps | 96 kbps |

- **F2.7.** При обрыве соединения приложение автоматически пытается переподключиться (минимум 3 попытки с увеличивающимся интервалом). Если все попытки безуспешны — показывается уведомление стримеру.

### 4.3 Оверлей с информацией о игре

Оверлей композитится поверх видео **на стороне стримящего устройства** до отправки потока в RTMP. Зритель не видит сырое видео — он видит готовое с плашками.

#### 4.3.1 Содержимое оверлея

- **F3.1. Статусы игроков** — основная плашка для каждого игрока. Содержит:
  - Имя игрока (никнейм)
  - Текущий статус: жив / убит / голосует / молчун / [другие статусы по факту существующих в Mafbase]
  - Роль игрока — отображается **только после раскрытия** (например, после смерти или конца игры)

  > **ДОПУЩЕНИЕ:** Конкретный набор статусов и правила раскрытия ролей нужно сверить с существующей моделью игры в Mafbase. В этом BRD используется обобщённый набор.

- **F3.2. Логотип турнира** — статичный, в углу экрана. Загружается из настроек турнира.
- **F3.3. Логотипы спонсоров** — опционально, ротация. До 3 логотипов, переключение каждые N секунд.

  > **ДОПУЩЕНИЕ:** Поддержка логотипов спонсоров — это требование от B2 (расширение возможностей для спонсоров). Если для MVP это излишне — выносим в v1.1.

- **F3.4. Текущий раунд / счёт** — небольшая плашка, показывает номер раунда (день/ночь) и общий счёт по столу/турниру.
- **F3.5. Имена игроков и роли** — после раскрытия (см. F3.1).

#### 4.3.2 Поведение оверлея

- **F3.6.** Оверлей ВСЕГДА присутствует на стриме (не отключается стримером во время игры).

  > **ДОПУЩЕНИЕ:** Возможность временно скрыть оверлей (например, для перерыва) можно добавить, но в MVP оставляем всегда видимым.

- **F3.7.** При изменении статуса игрока на стриме плашка обновляется с **анимацией перехода** (плавное изменение цвета/иконки, ~300мс), чтобы зритель видел изменение.
- **F3.8.** В моменты без изменений оверлей статичен и не потребляет процессор/батарею для перерисовки.
- **F3.9.** Оверлей рендерится в композитном слое поверх видео с поддержкой полупрозрачности.

### 4.4 Синхронизация состояния с ведущим

- **F4.1.** Изменения статусов, внесённые ведущим в его приложении, передаются на сервер Mafbase.
- **F4.2.** Стримящее устройство получает изменения с сервера в реальном времени и обновляет оверлей.
- **F4.3.** Время задержки между действием ведущего и обновлением оверлея на стриме — не более **2 секунд** при стабильной сети.

  > **ДОПУЩЕНИЕ:** 2 секунды — компромисс между сложностью реализации (WebSocket vs polling) и UX. Для зрителя это незаметно, потому что в RTMP уже есть свои несколько секунд latency.

- **F4.4.** При временной потере связи с сервером оверлей продолжает показывать последнее известное состояние. После восстановления связи — синхронизируется.
- **F4.5.** Изменения, сделанные оффлайн ведущим, применяются по порядку при восстановлении связи (через очередь действий с timestamps).

### 4.5 Локальная запись

- **F5.1.** Опционально (выбирается стримером перед стартом) — параллельно с RTMP-стримом записывать видео в локальный файл на устройстве.
- **F5.2.** Запись содержит **ровно тот же контент**, что и стрим (т.е. с оверлеем).
- **F5.3.** Формат записи — MP4 (H.264 + AAC).
- **F5.4.** После завершения стрима пользователю показывается информация о сохранённом файле и возможность поделиться им через системный share sheet.

  > **ДОПУЩЕНИЕ:** Для MVP — простая запись в публичную папку приложения. Загрузка записи на внешние сервисы (YouTube, облако) — за рамками MVP.

### 4.6 Управление активной трансляцией

- **F6.1.** Во время стрима на экране стримера отображается:
  - Превью композитной картинки (камера + оверлей) — то, что видят зрители
  - Длительность трансляции
  - Индикатор сети (зелёный/жёлтый/красный)
  - Текущий битрейт
  - Индикатор уровня микрофона
  - Кнопки: пауза/возобновление, остановка стрима, переключение фронт/тыл камеры
- **F6.2.** Стример может **остановить** трансляцию в любой момент. После остановки — экран с итогами (длительность, средний битрейт, путь к локальному файлу если запись была включена).
- **F6.3.** При случайном уходе приложения в фон стрим продолжается (через foreground service на Android, background mode на iOS) до тех пор, пока ОС не убъёт приложение или пользователь не закроет его явно.

  > **ДОПУЩЕНИЕ:** На iOS видеозахват в бэкграунде запрещён системой — при сворачивании приложения камера выключается, стрим продолжается со «statics» картинкой или прерывается. Точное поведение нужно зафиксировать с учётом ограничений iOS.

### 4.7 Доступ к фиче (монетизация)

- **F7.1.** Фича доступна только для **оплаченных турниров**.

  > **ДОПУЩЕНИЕ:** Логика монетизации: фича включена в стоимость турнира без дополнительных лимитов. Если у Mafbase сложнее модель (тарифные планы организаторов) — нужно адаптировать.

- **F7.2.** Если организатор турнира не оплатил — пункт меню «Запустить трансляцию» виден, но при попытке запуска показывает экран с информацией об оплате.
- **F7.3.** Нет ограничений на количество стримов, длительность одной сессии или общий час трансляций в рамках одного турнира.

## 5. Нефункциональные требования

### 5.1 Производительность

- **NF1.1.** Composite + encoding pipeline должен стабильно держать 30 FPS при разрешении 720p на устройствах референсного уровня.

  > **ДОПУЩЕНИЕ:** Референсный уровень устройств: Android — Snapdragon 7-серии и выше, 4+ ГБ RAM, не старше 4 лет. iOS — iPhone XS и новее. На устройствах ниже минимума фича работает, но без гарантии 30 FPS.

- **NF1.2.** Задержка от изменения статуса ведущим до обновления оверлея на стриме — не более 2 секунд (см. F4.3).
- **NF1.3.** Расход батареи: телефон при полностью заряженной батарее должен выдерживать минимум 3 часа непрерывного стрима в Medium preset.

  > **ДОПУЩЕНИЕ:** В реальном сценарии турнир идёт несколько часов, и стример скорее всего поставит телефон на зарядку через USB. Тем не менее 3 часа — разумный baseline для случая обрыва зарядки.

- **NF1.4.** Нагрев устройства не должен приводить к thermal throttling в течение часа при normal ambient temperature (~22°C).

### 5.2 Стабильность

- **NF2.1.** Стрим-сессия длительностью 4 часа должна завершаться без необходимости перезапуска (отсутствие memory leaks, native crashes).
- **NF2.2.** При потере сетевого соединения приложение не должно падать; пытается переподключиться (см. F2.7).
- **NF2.3.** Приложение должно корректно обрабатывать поворот устройства, входящий звонок, понижение системой яркости/громкости — без обрыва стрима.

### 5.3 Поддержка устройств

- **NF3.1.** **Android:** API 26 (Android 8.0) и выше.

  > **ДОПУЩЕНИЕ:** API 26 — типичный минимум для современных Flutter-приложений. Можно поднять до API 28+, если у Mafbase уже более высокий минимум.

- **NF3.2.** **iOS:** iOS 14.0 и выше.

  > **ДОПУЩЕНИЕ:** iOS 14 — также типичный минимум. Уточнить с действующим минимумом Mafbase.

- **NF3.3.** Поддержка обеих ориентаций (landscape основная для стрима, portrait для UI настройки).

### 5.4 Безопасность и приватность

- **NF4.1.** Stream keys хранятся в защищённом хранилище ОС (Android Keystore / iOS Keychain), не в SharedPreferences/UserDefaults.
- **NF4.2.** Контент стрима не сохраняется на серверах Mafbase (если только не запрошена опция облачной архивной записи — за рамками MVP).
- **NF4.3.** Перед началом стрима — явное согласие пользователя на трансляцию изображений участников турнира (через стандартный consent flow).

  > **ДОПУЩЕНИЕ:** Юридическая модель согласий участников турнира на трансляцию (включая ВКЛЮЧАЯ публикацию имени и роли в Мафии) должна быть проработана отдельно. На уровне приложения предполагается, что согласие участников турнира получено организатором до старта.

### 5.5 Локализация

- **NF5.1.** Поддержка русского и английского языков (базовый набор Mafbase).

  > **ДОПУЩЕНИЕ:** Если Mafbase сейчас только на русском — оставляем только русский в MVP.

## 6. Out of Scope (что НЕ делаем в MVP)

Чтобы сфокусировать MVP, эксплицитно фиксируем что **не входит** в первую версию:

- Multi-source стриминг (несколько камер, картинка-в-картинке)
- Стрим экрана устройства вместо камеры
- Запись стрима на серверы Mafbase / облачное хранение
- Сложные сцены и переходы (как в OBS): чёрный экран, BRB-экран, intro/outro
- Модерация чата зрителей внутри приложения
- Trim/edit сохранённой локальной записи
- Поддержка SRT, HLS-публикация, WebRTC — только RTMP/RTMPS
- Внешние микрофоны / мониторинг звука через наушники
- Управление статусами с самого стримящего устройства (только с отдельного устройства ведущего, см. 3.2)
- Поддержка планшетов как стримящего устройства (только телефоны)
- Кастомизация дизайна оверлея организатором (цвета, шрифты, расположение)

  > **ДОПУЩЕНИЕ:** Кастомизация дизайна оверлея — крайне желательная фича для брендинга турниров и спонсоров, но усложняет MVP. Предлагаю отложить в v1.1.

## 7. Технический подход (high-level)

Подробная техническая архитектура — отдельный документ (TDD). Здесь — высокоуровневое описание для понимания scope.

### 7.1 Архитектура

- Фича реализуется как **Flutter-плагин** (`mafbase_stream`), интегрируемый в существующее приложение Mafbase.
- Внутри плагина:
  - **C++ ядро** — кросс-платформенный модуль, содержащий: GL композитор камеры и оверлея, RTMP-клиент, контроллер адаптивного битрейта, state machine стрим-сессии. Подключается через JNI (Android) и Obj-C++ bridge (iOS).
  - **Платформенные слои:**
    - Android: Kotlin + Camera2 + MediaCodec + foreground service. Запускает отдельную нативную `Activity` для экрана стрима.
    - iOS: Swift + AVFoundation + VideoToolbox. Открывает нативный `UIViewController` modally.
- Flutter-сторона (Dart API плагина) — только тонкая обёртка для:
  - Запуска нативного экрана стрима с параметрами
  - Получения событий о состоянии стрима (started, stopped, error, stats)
  - Передачи обновлений состояния игры из существующего state management Mafbase в нативный оверлей через MethodChannel/EventChannel
- Кадры видео НЕ пересекают границу Dart ↔ натив. Только маленькие state-апдейты.

### 7.2 Ключевые библиотеки/технологии

- **RTMP-клиент:** на старте — FFmpeg (libavformat) для скорости разработки. С возможностью замены на собственную реализацию или libsrt позже.
- **Композитинг:** OpenGL ES 3.0 на обеих платформах (для единообразия шейдерного кода).
- **Энкодинг:** аппаратный — MediaCodec (Android), VideoToolbox (iOS).
- **Real-time синхронизация ведущий ↔ стример:** через существующий бэкенд Mafbase. Предположительно WebSocket или Server-Sent Events.

  > **ДОПУЩЕНИЕ:** Конкретный механизм real-time коммуникации зависит от уже имеющегося в Mafbase. Если есть готовый канал — используем его. Если нет — закладываем разработку.

### 7.3 Тестируемость

- C++ ядро тестируется unit-тестами на десктопе (Linux/macOS) через gtest, без эмуляторов.
- Плагин имеет собственное `example/` Flutter-приложение для интеграционных тестов и ручной проверки.
- E2E-тесты стрим-pipeline проводятся против локального nginx-rtmp.

## 11. Глоссарий

- **RTMP** — Real-Time Messaging Protocol, протокол передачи видеопотоков, стандарт индустрии для стриминга.
- **RTMPS** — RTMP поверх TLS (защищённое соединение).
- **Stream key** — секретный ключ, идентифицирующий канал на принимающем сервере (например, на VK Видео или YouTube).
- **Оверлей** — графический слой, накладываемый поверх видео (плашки, логотипы и т.д.).
- **Композитинг** — процесс объединения нескольких визуальных слоёв в одну картинку.
- **Адаптивный битрейт** — динамическое изменение качества потока в зависимости от условий сети.
- **MediaCodec / VideoToolbox** — системные API аппаратного видеокодирования на Android и iOS соответственно.
- **OBS** — Open Broadcaster Software, популярная десктопная программа для стриминга, де-факто стандарт референса для UX.
